## 요약
이 백서는 [[AI/Engineering/Agent_Engineering/Agent_Engineering|Agent Engineering]]를 "외부 정보 접근과 추론·논리를 결합한 GenAI 모델 기반 시스템"으로 정의한다. 모델 단독으로는 정적 지식과 단발 추론에만 갇혀 있지만, Agent는 도구를 통해 외부 세계와 상호작용하고 세션 히스토리를 관리하며 ReAct·CoT 같은 인지 아키텍처(Cognitive Architecture)를 통해 다단계 작업을 수행한다는 것이 핵심 메시지다.

## 핵심 정의
- **Agent**: 환경을 관찰하고 도구로 행동하여 목표를 자율적으로 달성하는 애플리케이션.
- **Cognitive Architecture**: Model + Tools + Orchestration Layer가 결합된 추론 패턴.

## Agent vs Model 비교
| 측면 | Models | Agents |
|---|---|---|
| 지식 | 학습 시점에 고정 | Tools를 통해 확장 |
| 추론 | 1회 inference | 세션 히스토리 + 다단계 루프 |
| 도구 사용 | 없음 | 네이티브 |
| 인지 아키텍처 | 없음 | CoT, ReAct, LangChain 등 |

## Cognitive Architecture (Reasoning 프레임워크)
- **ReAct** ([[AI/Engineering/Agent_Engineering/Planning_and_Reflection|Planning & Reflection]]): Thought → Action → Action Input → Observation → Final Answer 루프. SOTA 능가 사례 다수.
- **Chain-of-Thought (CoT)**: 하위 기법 — self-consistency, active-prompt, multimodal CoT.
- **Tree-of-Thoughts (ToT)**: lookahead와 다중 추론 경로 탐색.
- ReAct walkthrough 예: 사용자 질의 → Question → Thought → Action(Flights/Search/Code/None) → Action Input → Observation → 반복 → Final Answer.

## Tools — 3가지 유형
### Extensions
- Agent와 API 사이의 다리. **Agent-side 실행**.
- Agent에게 예제와 파라미터 명세로 사용법을 가르친다.
- 1-to-many: 하나의 Agent가 여러 Extension/API를 다룰 수 있다.
- 예: `Extension.from_hub("code_interpreter")` → `generate_and_execute` operation 호출.

### Functions (Function Calling)
- 자기 완결 코드 모듈. **Client-side 실행**.
- 모델은 함수명+인자만 출력, 실제 호출은 클라이언트가 수행 ("API stubbing").
- 사용 시점: API 미들웨어 필요, 보안 제약, 배치/HITL 타이밍, 후처리 필요.
- 코드 예: `gemini-2.0-flash-001`, `automatic_function_calling=disabled`, `tool_config=ToolConfig(function_calling_config=FunctionCallingConfig(mode='ANY'))`.
- `display_cities(cities, preferences)` 함수가 `{cities: ["Aspen", "Park City", "Whistler"], preferences: "skiing"}` JSON 반환.

### Data Stores
- 벡터 데이터베이스 기반 실시간 데이터. PDF·스프레드시트·웹사이트·HTML·구조화/비구조화 원본 형식 그대로 활용.
- [[AI/sources/whitepaper_emebddings_vectorstores_v2|Embeddings & Vector Stores]]를 통해 연결.

### RAG 라이프사이클
사용자 쿼리 → embedding 모델 → vector DB에서 **ScaNN** 알고리즘으로 매칭 → 텍스트 검색 → Agent가 응답 형성 ([[AI/Engineering/Context_Engineering/Retrieval_Strategies/RAG/RAG|RAG]]).

### Tools 비교 표
| Tool | 실행 위치 | 대표 용도 |
|---|---|---|
| Extensions | Agent-side | 멀티홉 API 계획 |
| Function Calling | Client-side | 보안/배치/비인터넷 API |
| Data Stores | Agent-side | RAG (웹/구조화/비구조화) |

## Targeted Learning
- **In-context Learning**: Prompt + tools + few-shot at inference (ReAct 예시).
- **Retrieval-based In-context Learning**: Vertex AI Example Store, RAG 기반 동적 예시.
- **Fine-tuning**: 특정 도메인을 위한 사전 학습 ([[AI/Engineering/Model_Engineering/Full_Fine-Tuning|Full Fine-Tuning]]).
- 셰프 비유: 주어진 레시피 / 잘 갖춰진 팬트리 / 요리 학교.

## LangChain Quickstart
- `langgraph.prebuilt.create_react_agent`, `SerpAPIWrapper`, `GooglePlacesTool`, `ChatVertexAI(model="gemini-2.0-flash-001")`.
- 다단계 쿼리: "지난주 Texas Longhorns가 누구와 경기했는가? 상대 팀 경기장 주소는?" → 검색 → places → "Sanford Stadium, 100 Sanford Dr, Athens, GA".

## 프로덕션 (Vertex AI)
- Vertex AI Agent Builder, Vertex Extensions, Vertex Function Calling, Vertex AI Example Store.
- 결론에서 "agent chaining", "mixture of agent experts" 패턴 언급.

## Key Takeaways
- Agent의 품질은 "모델의 추론·행동 능력"과 "도구가 얼마나 잘 정의되었는가"에 비례한다.
- 도구는 Extensions·Functions·Data Stores의 3가지 큰 분류로 정리된다.
- 학습 전략은 in-context → retrieval-based → fine-tuning 순으로 비용/깊이가 증가한다.
