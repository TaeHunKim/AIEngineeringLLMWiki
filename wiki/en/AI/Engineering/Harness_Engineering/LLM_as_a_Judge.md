# LLM-as-a-Judge

## Overview

**LLM-as-a-Judge** is a paradigm that uses a powerful LLM (judge model) to automatically evaluate the outputs of other LLMs, instead of having humans evaluate directly. It achieves reasonable evaluation quality while solving the cost and speed problems of human evaluation.

## Origin

- **Authors**: Zheng et al. (2023)
- **Paper**: "Judging LLM-as-a-Judge with MT-Bench and Chatbot Arena" — [arXiv:2306.05685](https://arxiv.org/pdf/2306.05685)
- **Key contribution**: Demonstrated that using GPT-4 as a judge approaches human evaluator agreement (80%+)

## MT-Bench

Multi-turn QA benchmark introduced by Zheng et al.:
- 8 categories × 10 questions = 80 high-quality multi-turn questions
- Covers math, reasoning, coding, writing, roleplay, etc.
- GPT-4 judge model scores 1–10

## Evaluation Methods

### 1. Single Output Grading

Evaluating one response on an absolute scale:
```python
grading_prompt = """
Rate the following response from 1 to 10, considering:
- Accuracy (30%)
- Completeness (25%)
- Clarity (25%)
- Helpfulness (20%)

Question: {question}
Response: {response}

Reasoning:
Score (1-10):
"""

score = judge_llm.invoke(grading_prompt)
```

### 2. Pairwise Comparison

Compare two responses and select the better one:
```python
pairwise_prompt = """
Evaluate which of the following two responses is better.

Question: {question}
Response A: {response_a}
Response B: {response_b}

Explain which response is better and why, then answer with "A wins", "B wins", or "Tie".

Evaluation:
Conclusion:
"""
```

### 3. Reference-based Grading

Compare against a ground truth (golden reference):
```python
reference_prompt = """
Evaluate how well the following response matches the correct answer.

Question: {question}
Reference answer: {reference}
Response to evaluate: {response}

Accuracy (0-10):
Completeness (0-10):
"""
```

## LLM-as-a-Judge Implementation (DeepEval)

```python
from deepeval import evaluate
from deepeval.metrics import GEval, AnswerRelevancyMetric, FaithfulnessMetric
from deepeval.test_case import LLMTestCase

answer_relevancy = AnswerRelevancyMetric(
    threshold=0.7, model="gpt-4o", include_reason=True
)

faithfulness = FaithfulnessMetric(threshold=0.8, model="gpt-4o")

correctness = GEval(
    name="Correctness",
    criteria="Evaluate whether the response is factually accurate",
    evaluation_steps=[
        "Identify each claim in the response",
        "Verify the factual accuracy of each claim",
        "Assign an overall accuracy score from 0-10"
    ],
    model="gpt-4o"
)

test_case = LLMTestCase(
    input="Where is Samsung Electronics headquartered?",
    actual_output="Samsung Electronics is headquartered in Suwon.",
    expected_output="Samsung Electronics is headquartered in Suwon, Gyeonggi-do, South Korea.",
    retrieval_context=["Samsung Electronics is headquartered in Suwon, Gyeonggi Province."]
)

evaluate([test_case], [answer_relevancy, faithfulness, correctness])
```

## LLM-as-a-Judge Biases

### 1. Position Bias

Same response gets different evaluation based on order. Swapping A→B order to B→A can change results.
```python
# Solution: evaluate twice with swapped order, then average
score_ab = judge(response_a, response_b)
score_ba = judge(response_b, response_a)
final_score = average(score_ab, 1 - score_ba)  # direction correction
```

### 2. Verbosity Bias

Tendency to prefer longer responses. Higher score for longer regardless of actual quality.
```python
# Solution: length normalization or explicit instruction
"Evaluate only the quality of content, not length. Treat concise answers equally."
```

### 3. Self-Enhancement Bias

Preference for responses generated by the same model. GPT-4 gives high scores to GPT-4 responses.
```python
# Solution: use different models (Anthropic evaluating OpenAI, etc.)
```

## Improving LLM-as-a-Judge Reliability

```python
# Multi-judge: evaluate with multiple models and aggregate
judges = [gpt4, claude, gemini]
scores = [judge.evaluate(response) for judge in judges]
```

## Agent-as-a-Judge

Extension of LLM-as-a-Judge specialized for evaluating agent systems. The core difference from LLM-as-a-Judge (which evaluates final text output) is that it **evaluates the entire execution trajectory**.

Details (original paper, Critic Agent pattern, Multi-Agent-as-Judge, Agent Simulation, pros/cons) → **[[en/AI/Engineering/Harness_Engineering/Agent_as_a_Judge|Agent-as-a-Judge]]**

## Role in AI Engineering

LLM-as-a-Judge is the **core tool of Evaluation Engineering**. It rapidly evaluates thousands of responses without human evaluators, and integrating into CI/CD pipelines enables automatic detection of quality regressions when models change. It forms the foundation of all LLMOps decisions — A/B testing, prompt optimization, model selection. In agent systems, it extends to [[en/AI/Engineering/Harness_Engineering/Agent_as_a_Judge|Agent-as-a-Judge]] for execution trajectory-level evaluation and pre-deployment stress testing.
