## Summary
Embeddings are a "lossy compression" mapping heterogeneous data (text, image, audio, video) into a shared vector space. Starting from the simple fact that BERT maps text to a 768-dimensional vector, it covers evaluation metrics (Precision@k, Recall@k, nDCG, BEIR/MTEB), the evolution of embedding models, ANN algorithms, vector DB suites, and comprehensive applications in RAG.

## Nature of Embeddings
- Semantic similarity = geometric distance. Analogy: Latitude/longitude is a 2D embedding of a location.
- Embeddings generated by different models are not compatible → consistent version is mandatory.
- **3-step retrieval**: (1) pre-calculate corpus embeddings, (2) embed the query, (3) NN search.

## Evaluation
- **Precision@k** = proportion of retrieved results that are relevant (e.g., 7/10 = 0.7)
- **Recall@k** = proportion of all relevant items that are retrieved
- **nDCG**: DCG = Σ rel_i / log2(i+1); normalized by ideal ordering (0.0–1.0)
- **Benchmarks**: **BEIR** (zero-shot retrieval), **MTEB** (Massive Text Embedding Benchmark)
- Tools: trec_eval, pytrec_eval
- Progress: BEIR ~10.6 for original BERT → current Google 2025 embedding average 55.7

## Tokenization
String → tokens (wordpiece/character/word/numbers/punct) → integer ID (within vocab_size range) → optional one-hot. Keras `Tokenizer` example.

## Word Embedding (Context-free)
- **Word2Vec** (Mikolov 2013) — CBOW (context → middle, excellent for frequent words) vs. Skip-gram (middle → context, excellent for rare words)
- **GloVe** (Pennington/Socher/Manning 2014) — global co-occurrence matrix factorization
- **SWIVEL** (Shazeer 2016) — distributed parallel matrix factorization, fast
- **FastText** — Word2Vec + subword

## Document Embedding
### Shallow / Bag-of-Words
- **LSA** (1990), **LDA** (2001), **TF-IDF / BM25** (strong baseline), **Doc2Vec** (Le & Mikolov 2014)
- Limitations: Ignores word order and deep semantics

### Deep / Pretrained
- **BERT** (Devlin 2018) — bidirectional encoder, MLM + NSP, [CLS] token = sentence vector
- **Sentence-BERT** (Reimers & Gurevych 2020), **SimCSE** (Gao 2021), **E5** (Wang 2022)
- **T5** (2019, 11B), **PaLM** (2022, 540B), **Gemini**, **GPT**, **LLaMA** — backbones
- **GTR**, **Sentence-T5** — outperforms BERT-family
- **Gemini-based Vertex Embeddings** — SOTA on public benchmarks
- **Matryoshka Embedding** (Kusupati 2022; Matryoshka Quantization 2025) — variable dimension → saves storage space
- **Multi-vector**: **ColBERT** (Khattab & Zaharia 2020), **XTR** (Lee 2023), **ColPali** (Faysse 2024 — processes documents as images without OCR)

## Image & Multimodal
- Image embedding: penultimate layer of CNN/ViT
- Multimodal: alignment into a single latent space. Vertex `multimodalembedding@001`
- ColPali: processes documents as rendered images

## Structured / Graph Embeddings
- General structured: dimensional reduction like PCA → anomaly detection
- User/item: recommendation (user and item vectors in one space)
- Graph: **DeepWalk, Node2Vec, LINE, GraphSAGE** (Hamilton 2017)

## Learning Embeddings
Dual-encoder dominant. **Contrastive loss**: <anchor, positive, [negatives]> — pulls positive closer and pushes negatives away. Fine-tuned after initialization from a foundation model (BERT/T5/GPT/Gemini/CoCa). Data: human-labeled, synthetic, distillation, hard-negative mining.

## Vector Search Algorithms
- **Similarity metrics**: Euclidean (low dimension), cosine (high dimension), dot product (equivalent to cosine when normalized)
- Brute force is O(N) — O(log N) using ANN (approximate NN) for millions or more

### LSH (Locality Sensitive Hashing)
A hash that maps similar items to the same bucket. Bucket + adjacent search. Analogy: ZIP code lookup.

### Tree-based
- **KD-tree** (Friedman 1977), **Ball-tree** (better suited for high dimensions)

### HNSW (Hierarchical Navigable Small World, Malkov & Yashunin 2016)
- Multi-layer proximity graph. Upper layers have long links, lower layers have short links.
- Greedy search starting from the top layer → local minimum at each layer.
- O(log N), can be combined with quantization.
- FAISS: `IndexHNSWFlat(d, M=32)`

### ScaNN (Google, Guo 2020)
- **anisotropic vector quantization** (outperforms PQ)
- Steps: (1) optional partitioning (heuristic ≈ √N), (2) top partition selection, (3) exact/approximate distance, (4) optional rescoring
- TF Recommenders: `tfrs.layers.factorized_top_k.ScaNN(distance_measure='dot_product', num_leaves=4, num_leaves_to_search=2)`

### FAISS (Facebook, Douze 2024)
Supports multiple indexes such as HNSW, PQ, etc.

## Vector Databases
- **Managed**: Google Cloud **Vertex AI Vector Search** (ScaNN-based, `MatchingEngineIndex.create_tree_ah_index(...)`), **AlloyDB** & **Cloud SQL Postgres** (pgvector + AlloyDB native ScaNN), **Elasticsearch**, **Pinecone** (HNSW), **Weaviate** (HNSW), **BigQuery vector search** (OLAP/batch)
- **OSS**: Weaviate, ChromaDB, pgvector
- **Hybrid search**: vector ANN + keyword + metadata filtering

## Operational Considerations
- Embeddings shift when retraining models → automatic re-embedding/purge required
- Embeddings are weak at literal/syntactic (ID/code) representations → combine with full-text search
- Workload: OLTP → AlloyDB/Spanner/Postgres/CloudSQL, OLAP → BigQuery vector search

## Applications
- Search (Q&A, recommendation), semantic text similarity, classification, clustering, reranking
- **Q&A with sources (RAG)**: retrieval → prompt expansion → grounded response. Reduces hallucination, secures freshness without retraining
- Code example: LangChain `RetrievalQA.from_chain_type(llm=VertexAI(model_name="gemini-pro"), chain_type="stuff", retriever=...)`. 768D `text-embedding-005`, `DOT_PRODUCT_DISTANCE`, `STREAM_UPDATE`

## Key Takeaways
- Select embedding models according to data distribution, fine-tune if necessary.
- ScaNN and HNSW are the standards for the speed/accuracy trade-off.
- Embeddings + Vector DB + ANN = the infrastructure substrate of search, recommendation, and RAG.
